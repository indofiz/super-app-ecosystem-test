# BFF Audit — 2026-05-13

> Scope: `bff/` only. Independent audit performed against the codebase as of 2026-05-13. Reviewer focus areas: **security**, **performance**, **maintainability**, **architecture**, **production-readiness**.
>
> Verdict in one line: **architecturally solid, security posture is strong after the 2026-05-10/11 work, but a handful of correctness, robustness, and ops items remain before this is "calling-from-home" production-grade.**
>
> This document is a *complement* to `bff/docs/IMPROVEMENT_PLAN.md`, not a replacement. Where an item overlaps, the IMPROVEMENT_PLAN section is referenced and the priority restated. New findings (not yet captured in the IMPROVEMENT_PLAN) are explicitly tagged **NEW**.

---

## 0. Executive Summary

What works well (do not touch):

- BFF-only auth model is implemented faithfully — the mobile app never sees Keycloak directly and never holds a refresh token.
- PKCE-on-PKCE flow (app PKCE wrapped by BFF PKCE) is correct.
- Single-use codes via Redis `GETDEL` (atomic) for both `authstate:*` and `bffcode:*`.
- RS256 internal JWT with kid-based rotation + JWKS endpoint — Kong only holds public keys; private signing key never leaves the BFF process.
- Bearer-bound `sid` on `/refresh`/`/logout`/`/me` (closed §2.1).
- KC tokens verified against the realm's JWKS (closed §2.3) — including the `azp` belt-and-braces check.
- Redis-backed rate limiting with per-endpoint prefixes and `/me` keyed on `sid` (closed §3.5).
- `prom-client` metrics with route allowlist (cardinality bounded) + OTEL tracing scaffolding + real `/healthz`/`/readyz`/`/livez` (closed §3.1).
- Token-redacting Pino logger with a defensive `redact.paths` list.
- Single-use `bffcode`, deeplink error redirection on KC errors, idempotent logout, `InvalidGrantError` purges sessions.
- Strong test posture for the happy and main rejection paths (59 tests).

Top issues, in priority order, are in §1. Detailed findings follow.

## 2. Detailed Findings

### 2.1 Security

#### S-1 [High, correctness] — `endSession` metric mislabels failures as success **NEW**

**File:** `src/lib/keycloak.ts:156-177`

```ts
async endSession(refreshToken: string): Promise<void> {
  ...
  await this.time('end_session', async () => {
    try {
      await this.http.post(disc.end_session_endpoint!, body, { ... });
    } catch (err) {
      this.opts.log.warn({ err }, 'keycloak end_session call failed');
    }
  });
}
```

The inner `try`/`catch` swallows the error before `time()` sees it, so `observe('end_session', 'ok', start)` runs on the failure path. Result: `keycloak_request_duration_seconds{op="end_session",outcome="error"}` will never increment in production, and a KC outage during logouts is invisible to SRE.

**Fix:** rethrow inside `time()` then catch at the outer scope, or pass the outcome explicitly:

```ts
await this.time('end_session', async () => {
  try {
    await this.http.post(disc.end_session_endpoint!, body, { ... });
  } catch (err) {
    this.opts.log.warn({ err }, 'keycloak end_session call failed');
    throw err;  // let time() observe outcome=error
  }
}).catch(() => {/* best-effort: caller already wiped session */});
```

#### S-2 [Medium] — `error_description` leaks zod issue text into 4xx responses **NEW**

**File:** `src/middleware/error.ts:28`, `src/auth/handlers/*.ts` (every `badRequest` call)

`badRequest('invalid_request', parsed.error.issues[0]?.message ?? 'Invalid body')` propagates zod's issue message verbatim. Zod messages can include user-supplied values for the failing field. Example: a bad `state=<x-with-special-chars>` becomes part of the public response body. Low-impact (no secret content) but it's an information leak and a parser-error oracle that helps attackers map the validator.

**Fix:** map zod issues to a fixed code/message vocabulary in handlers; reserve `error_description` for stable strings; log the raw issue server-side only. Alternatively redact `error_description` in `NODE_ENV=production` and rely on the stable `error` code.

#### S-3 [Medium] — `app.set('trust proxy', 1)` is hardcoded *(plan §2.7)*

**File:** `src/app.ts:30`

`trust proxy = 1` trusts exactly one hop (nginx today). The day someone fronts the BFF with a second LB (Cloudflare, internal mesh sidecar), `X-Forwarded-For` becomes spoofable end-to-end, breaking rate-limit keying and IP logging silently.

**Fix:** env-driven `TRUST_PROXY` (CSV of CIDRs or one of `loopback,linklocal,uniquelocal`) passed to `app.set('trust proxy', ...)`.

#### S-4 [Medium] — OIDC discovery cached forever *(plan §2.10)*

**File:** `src/lib/keycloak.ts:84-104`

`this.discovery` is set once and never refreshed. Issues:

- KC config drift (jwks_uri move, endpoint version bump) only takes effect after a BFF restart.
- The `jose.createRemoteJWKSet` in `keycloakJwt.ts` is built from the cached `jwks_uri` (see `getJwks()` at `src/lib/keycloakJwt.ts:91-108`), so it inherits the same staleness.

**Fix:** soft TTL (e.g., 60 min) with background refresh; explicit invalidation on a verification path that sees a `kid` not in the JWKS.

#### S-5 [Medium] — No back-channel logout *(plan §2.5)*

If the user logs out via web/kiosk against KC directly, the BFF's `session:*` row stays alive. Bounded by the 30d TTL but a real correctness gap.

**Fix:** `POST /auth/back-channel-logout` validates a KC `logout_token` against JWKS, then deletes matching sessions. Maintain a `sub → [sid,...]` index in Redis (`SADD`/`SREM` on create/delete) for O(1) revocation per user.

#### S-6 [Medium] — `verifyAllowingExpired` widens *all* time-claim tolerances **NEW**

**File:** `src/lib/internalJwt.ts:133-138`

```ts
async verifyAllowingExpired(token, graceSeconds = 24 * 60 * 60) {
  return this.verifyInternal(token, { clockTolerance: graceSeconds });
}
```

`jwtVerify`'s `clockTolerance` is applied to `exp`, `nbf`, and `iat`. So a token with `nbf` 12 hours in the future (e.g., a BFF whose clock got bumped forward, was caught + corrected, then signed tokens during the bad window) is accepted as "valid but expired" for the next 24h. Mostly theoretical because we set `nbf=0s` and KC enforces NTP, but the contract is wider than the intent.

**Fix:** Use a tighter `clockTolerance` (e.g., 60s for skew) and instead bypass `exp` checking with jose's `currentDate` override or by catching the `JWTExpired` error specifically:

```ts
try {
  return await jwtVerify(token, key, { issuer, audience, algorithms: [alg], clockTolerance: 60 });
} catch (err) {
  if (err instanceof errors.JWTExpired && withinGrace(err.payload.exp, graceSeconds)) {
    return { payload: err.payload, ... };
  }
  throw err;
}
```

#### S-7 [Low] — Pino redaction wildcards do not match nested *(plan §2.8)*

**File:** `src/lib/logger.ts:8-26`

`'*.access_token'` matches `{ access_token }` and `{ x: { access_token } }` only at the documented depth. Pino-http logging adds `req.body.X` and `res.body.X`; verify a deeply nested response payload still gets scrubbed.

**Fix:** add explicit nested paths (`req.body.tokens.access_token`, `res.headers['set-cookie']`, etc.) and a vitest case that logs through pino-http and asserts the secret is absent.

#### S-8 [Low] — No replay protection on internal JWT *(plan §2.9)*

**File:** `src/lib/internalJwt.ts:118`

`jti` is set, never tracked. Acceptable at 5-min TTL. **Action:** add a one-paragraph "we do not denylist `jti` at TTL=5min" note in `bff/README.md` so any TTL-bump PR has to revisit this.

#### S-9 [Low] — KC client uses `client_secret` in body (RFC prefers Basic) **NEW**

**File:** `src/lib/keycloak.ts:112-118`, `:125-130`, `:162-166`

OAuth 2.0 says HTTP Basic is preferred over `client_secret_post`. Functional today (KC accepts both). Cosmetic — file as a follow-up.

#### S-10 [Low] — `/auth/callback` consumes `authstate` before id_token verification can fail **NEW**

**File:** `src/auth/handlers/callback.ts:38`

`authStateStore.take()` consumes the row via `GETDEL`. If subsequent `id_token` verification fails (e.g., transient JWKS fetch error), the user gets a deeplink `server_error` and **cannot retry that authorize flow** — the state is gone, they have to start over.

This is by design (single-use state). Worth documenting:

- in `bff/README.md` under "Failure modes"
- as a metric label so transient-vs-permanent failures are distinguishable on the dashboard.

### 2.2 Performance

#### P-1 [Low] — Discovery race on first request *(plan §5.2)*

Already covered.

#### P-2 [Low] — Stores `JSON.parse(raw) as T` with no schema check **NEW**

**Files:** `src/auth/stores/{authState,bffCode,session}.store.ts`

If a record's schema drifts between two deploys (added field A, removed field B), or a record was poisoned by a manual `redis-cli SET`, `JSON.parse` succeeds but `record.refreshToken` is `undefined` and the handler crashes deep with `TypeError: Cannot read properties of undefined`. The user sees a generic 500.

**Fix:** define a zod schema next to each `*Record` interface; `parse()` on read. Errors bubble as a clean `502 store_corrupted` (or `401 invalid_session`).

```ts
const SessionRecordZ = z.object({ refreshToken: z.string(), sub: z.string(), ... });
async get(sid: string): Promise<SessionRecord | null> {
  const raw = await this.redis.get(key(sid));
  if (!raw) return null;
  return SessionRecordZ.parse(JSON.parse(raw));
}
```

Pairs naturally with **§4.1** in the IMPROVEMENT_PLAN (extract `RedisJsonStore<T>`).

#### P-3 [Low] — KC HTTP client has no connection pooling / keepalive **NEW**

**File:** `src/lib/keycloak.ts:54-60`

Default `axios` agent. Under load, every KC call pays TCP+TLS handshake. For an auth BFF this is negligible at small scale; at higher RPS it dominates.

**Fix:** `new http.Agent({ keepAlive: true, maxSockets: 50 })` (and `https.Agent` for TLS). Cheap, no code-shape change.

#### P-4 [Low] — `verifyAndExtractProfile` re-verifies tokens already verified at `/callback`

`callback.ts` verifies id_token, persists to Redis, then `/token` reads + re-verifies. This is intentional belt-and-braces but adds 1–2 ms per `/auth/token` call. Acceptable today. If JWKS resolution ever has a tail of slow fetches, this becomes more visible.

### 2.3 Maintainability

#### M-1 [Medium] — `verifyAndExtractProfile` lives in `handlers/token.ts` but is imported by `handlers/refresh.ts` **NEW**

**File:** `src/auth/handlers/token.ts:39-56`, imported at `src/auth/handlers/refresh.ts:12`

Cross-handler imports are a layering smell. A third caller would force a refactor. Move to `src/lib/keycloakJwt.ts` (or a new `src/auth/verifyProfile.ts`) so handlers depend on the same library symbol.

#### M-2 [Medium] — No zod validation of store reads *(see P-2)*

Same finding, framed as maintainability: schema-on-the-boundary is the standard idiom; absence is felt at every refactor.

#### M-3 [Low] — `req.id` augmentation uses cast *(plan §4.2)*

**File:** `src/middleware/requestId.ts:10`

`(req as unknown as { id: string }).id = id;` — replace with `declare module 'express-serve-static-core' { interface Request { id: string } }` in a `src/types/express.d.ts` (already done for `claims` in `src/middleware/sessionAuth.ts:8-12` — follow the same pattern).

#### M-4 [Low] — `SessionStore.update` is a redundant alias for `put` **NEW**

**File:** `src/auth/stores/session.store.ts:36-39`

The method literally calls `this.put(...)`. Either:

- Inline `put` at the call sites and delete `update`; or
- Make `update` add an `EXPIRE` semantics that `put` does not (would also address §5.1 sliding-TTL).

#### M-5 [Low] — `requireSessionBearer` collapses verify-failure reasons **NEW**

**File:** `src/middleware/sessionAuth.ts:38-45`

```ts
try { claims = await ... } catch { throw unauthorized('invalid_token', 'Bearer verification failed'); }
```

Good for security (no oracle). Bad for ops: every JWT problem looks identical in logs. Add a `log.debug({ err, kid: ..., reason: err.message }, 'bearer verify failed')` before rethrowing so support can correlate without exposing detail to the wire.

#### M-6 [Low] — ESLint config is minimal *(plan §4.4)*

Add `@typescript-eslint/no-floating-promises`, `@typescript-eslint/no-misused-promises`, `import/order`, `eslint-plugin-n` for NodeNext ext correctness.

#### M-7 [Low] — Test gaps *(plan §4.6)*

The plan already enumerates these. Adding a coverage gate (`vitest --coverage` with lines ≥ 80, branches ≥ 70) is the cheapest win.

#### M-8 [Low] — No OpenAPI / contract artifact *(plan §4.8)*

Generate from zod with `zod-to-openapi` and serve under `/openapi.json` (gated). Mobile + future web admin can codegen clients.

### 2.4 Architecture

#### A-1 [Low] — Stores duplicate boilerplate *(plan §4.1)*

3 files × `JSON.stringify` + `SET EX` + `GETDEL` + `JSON.parse`. Extract `RedisJsonStore<T>` with optional zod schema. Pairs with §P-2/M-2.

#### A-2 [Low] — Handler-level orchestration in `token.ts`/`refresh.ts` is non-trivial

Not a problem at the current scope. If a second SSO realm or a social-login provider lands, split into `auth/services/*` to keep handlers thin.

#### A-3 [Low] — Sub-5-minute revocation lag at Kong is a known accepted limitation *(plan §5.6)*

Document explicitly in the dashboard / runbook so on-call doesn't expect immediate revocation when they hit `redis-cli DEL session:*`.

#### A-4 [Low] — Dynamic-import bootstrap in `index.ts` is correct but subtle

The dance at `src/index.ts:16-22` (start tracing, then `await import(...)`) is required for OTEL auto-instrumentations to patch http/express/axios prototypes in time. Already commented but the comment could call out the alternative ("if you import a module statically at the top, instrumentation will be a no-op for that module"). One-liner improvement.

### 2.5 Production Readiness

#### PR-1 [High] — Graceful shutdown is broken *(plan §3.2)*

**File:** `src/index.ts:76-82`

```ts
const shutdown = async (signal: string) => {
  log.info({ signal }, 'shutting down');
  server.close(() => log.info('http server closed'));   // fire-and-forget
  redis.disconnect();                                   // yanks the socket
  await stopTracing();
  setTimeout(() => process.exit(0), 1000).unref();      // way too tight
};
```

Concrete fix (also flushes the logger, switches to `quit()`):

```ts
const shutdown = async (signal: string) => {
  log.info({ signal }, 'shutting down');
  server.closeIdleConnections?.();
  await new Promise<void>((r) => server.close(() => r()));
  try { await redis.quit(); } catch { redis.disconnect(); }
  await stopTracing();
  await new Promise<void>((r) => log.flush(r));
  process.exit(0);
};
const force = setTimeout(() => process.exit(1), 25_000).unref();
process.on('SIGTERM', () => void shutdown('SIGTERM').finally(() => clearTimeout(force)));
process.on('SIGINT', () => void shutdown('SIGINT').finally(() => clearTimeout(force)));
```

#### PR-2 [Medium] — Dockerfile hardening *(plan §3.4)*

**File:** `bff/Dockerfile`

Missing:

- `HEALTHCHECK CMD wget -qO- http://localhost:3000/readyz || exit 1` (or curl, or a tiny node script).
- `tini` (or `node --init` once available) for PID-1 signal forwarding.
- Drop write-permissions on `/app`.
- Decide on source-maps in prod: today `tsconfig.json` emits `sourceMap: true` — either ship and decode in logs (set `NODE_OPTIONS=--enable-source-maps`) or set `sourceMap: false` for the prod build.

#### PR-3 [Medium] — KC HTTP client hardening *(plan §3.6)*

**File:** `src/lib/keycloak.ts`

- Retries with jitter on `getDiscovery` only (3 retries, 5s budget).
- `maxContentLength` / `maxBodyLength` set to 64 KB.
- Outbound `x-request-id` carried from the inbound request (plumb `req.id` through, or use AsyncLocalStorage).
- `zod`-parse the KC token response shape before trusting `expires_in`/`token_type` etc.

Sketch:

```ts
const TokenZ = z.object({
  access_token: z.string(),
  id_token: z.string().optional(),
  refresh_token: z.string().optional(),
  token_type: z.string(),
  expires_in: z.number(),
  refresh_expires_in: z.number().optional(),
  scope: z.string().optional(),
});
```

#### PR-4 [Medium] — No per-route timeout *(plan §5.5)*

If KC stalls, a request sits there until axios's 10s timeout, then the inbound request resumes. Add an Express-level guard so a slow upstream returns a clean `504` rather than hanging the connection.

#### PR-5 [Medium] — No CI *(plan §3.7)*

Minimum:

- `npm ci && npm run typecheck && npm run lint && npm test -- --coverage`
- `npm audit --audit-level=high --omit=dev`
- Docker build + Trivy scan
- (Optional) `osv-scanner` for transitive deps

#### PR-6 [Low] — `express.urlencoded` is unused *(plan §5.4)*

Drop. Reduces parsing surface to zero risk; trivial.

#### PR-7 [Low] — Env TTL ceilings **NEW**

A typo (`SESSION_TTL_SECONDS=25920000`) silently becomes an 8-month session. Bound: e.g., `z.coerce.number().int().positive().max(60*60*24*90)` for session, `max(60*60)` for bffcode/authstate.

#### PR-8 [Low] — Log flush on shutdown — covered by PR-1.

#### PR-9 [Low] — `app.disable('etag')` for an API

JSON 200s do not benefit from etags here. Cheap to disable for predictable response headers.
